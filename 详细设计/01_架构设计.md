## 1. 架构概述

### 1.1 设计目标

VoiceGrow 是一个基于小爱音箱的儿童语音交互学习平台，采用服务端 ASR 模式，提供故事、音乐、英语学习和智能对话功能。

**核心设计原则：**
- **模块化**: 各组件独立开发、部署和扩展
- **高可用**: 服务降级策略，保证基本功能可用
- **低延迟**: 端到端响应时间 < 3秒
- **安全性**: 儿童内容安全过滤，数据隐私保护

### 1.2 技术选型

| 组件 | 技术选型 | 选型理由 |
|-----|---------|---------|
| 运行时 | Python 3.10+ | 生态丰富，AI 库支持完善 |
| Web 框架 | FastAPI | 高性能异步，WebSocket 原生支持 |
| ASR | faster-whisper | 本地部署，隐私保护，精度高 |
| TTS | ai-manager TTS API | 基于 Google Cloud TTS，音质好，永久缓存 |
| LLM | ai-manager AI API | 多模型支持 (Gemini/GPT/DeepSeek/Claude)，智能路由 |
| 数据库 | MySQL 8.0 | 成熟稳定，关系型数据存储 |
| 对象存储 | MinIO | S3 兼容，自托管，成本低 |
| 缓存 | Redis | 会话缓存，高性能 |
| 容器化 | Docker Compose | 简化部署，环境一致 |

## 4. 关键设计决策

### 4.1 服务端 ASR vs 客户端 ASR

| 方案 | 优点 | 缺点 |
|-----|-----|-----|
| **服务端 ASR (选用)** | 识别精度高、可定制、隐私可控 | 需要传输音频、服务器计算负载 |
| 客户端 ASR | 响应快、省带宽 | 精度受限、难以定制 |

**决策：采用服务端 ASR**
- 使用 faster-whisper，识别精度显著优于设备端
- 音频数据仅在内部网络传输，隐私风险可控
- 可根据场景定制识别参数

### 4.2 TTS 方案选择

| 方案 | 优点 | 缺点 |
|-----|-----|-----|
| 设备内置 TTS | 响应快、无网络依赖 | 音质一般、音色单一 |
| **ai-manager TTS API (选用)** | Google Cloud TTS 音质好、永久缓存、多账户轮转 | 需要网络 |
| Edge TTS | 免费、音质好 | 不稳定、无 SLA |
| Azure TTS | 音质好、多音色 | 成本高、无缓存机制 |

**决策：采用 ai-manager TTS API**
- 基于 Google Cloud TTS，音质显著优于设备内置 TTS
- 支持中英文多音色，适合儿童场景
- 永久缓存机制，相同文本仅合成一次，大幅降低成本
- 多账户自动轮转，避免配额限制

## 6. 性能设计

### 6.1 性能目标

| 指标 | 目标值 | 测量方法 |
|-----|-------|---------|
| ASR 延迟 | < 2s | 从音频结束到文本返回 |
| TTS 延迟 | < 1s | 从文本输入到音频开始播放 |
| 端到端延迟 | < 3s | 从用户说完到音箱开始响应 |
| 并发连接 | 100+ | 同时在线设备数 |
| 可用性 | 99.5% | 月可用时间占比 |

### 6.2 性能优化策略


## 7. 安全设计

### 7.1 安全威胁分析

| 威胁 | 风险等级 | 防护措施 |
|-----|---------|---------|
| 音频数据泄露 | 中 | 本地 ASR、传输加密 |
| 不当内容 | 高 | LLM 内容过滤、敏感词库 |
| 未授权访问 | 中 | 设备认证、API 鉴权 |
| DoS 攻击 | 低 | 连接限制、请求限流 |

### 7.3 数据隐私

- **音频数据**: 不存储原始音频，仅用于实时处理
- **对话历史**: 仅保留必要上下文，定期清理
- **设备信息**: 脱敏存储，不关联真实用户身份
- **日志脱敏**: 敏感字段哈希或掩码处理
